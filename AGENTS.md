# MetaEditor Rules
**最高原则**：功能相同的情况下，更短更精炼的代码就是更好的
新增代码前，先优先考虑能否用现有 utility 组合完成；如果需要引入新 utility，先判断它是否应当抽成可复用工具，避免在局部逻辑里重复造新层
新增概念前，先判断它是不是旧概念的变体、重复包装或临时残留；能合并就合并，能删就删，避免在状态、协议和 runtime 里留下并行冗余语义
同一件事只保留一条主路径，不要为了兼容或局部方便再留一个平行入口、隐藏入口或兜底分支
任何新状态、新队列、新缓存、新 flag，新增前先明确它的创建、消费、清理、重连和停机行为；如果讲不清，默认先不要加
必须严格遵守这些原则
## 协作规则
- 始终用中文回复用户
- 只改用户明确要求修改的内容，不顺手重排、重命名、整理或重构
- 先保持现有文件的顺序、排版、紧凑度和命名风格，再考虑别的优化
- 修改文件时，默认采用最小补丁，不整文件重写
- 注释默认用中文
- 禁止向项目目录及其任何子目录产生/遗留任何临时文件（如 debug_*.js, *.log, *.png 等）
- 所有的测试产物（截图、日志）必须输出到系统的临时目录（如 $env:TEMP），读取后立即清理
- 除非用户明确要求，禁止自己进行 commit
- 禁止使用这些表达（前面已经用了的就算了，从看到这条开始算）：`不是……而是……、而不是……、不再是……、到这里、这一轮、这条线、后面又、这里最后、收口、压回、钉死、兜底、落盘、闭环、说穿、能吃、一刀`
- 聊天/文档/代码里都不应该出现绝对路径（特别是md文件链接，这个东西本身就不该写，禁止写md文件链接）
- 测试里禁止写任何无超时等待，具体参见各个测试的具体等待方式
## 通用代码风格
- **缩进**：使用 **2 个空格**
- **行宽**：每行最多 **100** 个字符
## MoonBit 代码风格
- **重要**：符号级的改动/重构，优先用 `moon-ide` 命令行工具，这即节约 token 又不会出错
- 禁止使用 moon format，和用户的目标风格极端不兼容
- 保持紧凑写法，不主动拉开空行
- 状态 `let ...` 放在文件最开头
- 类型定义放在下面，并和对应的类型操作放在一起
- **类型标注写成 `x: T`，不要写成 `x : T`**
- 构造函数需要写双层大括号（函数块和 object 字面量），应该把两层大括号当成同一个符号来写，不要拆开整理，像这样：
  ```moonbit
  fn ... {{
    ...
  }}
  ```
- try catch 前面只有一个表达式时省略 try，直接 expr catch {…}
- 除非用户明确要求，不主动把短写法改成长写法
- 测试顶层不需要 catch ，除非是测试语义需要
## JavaScript 代码风格
- **不要分号**：语句结尾不要写分号
- **引号**：字符串使用单引号 `'`
- **大括号**：
  **最重要/最容易出错/要和行宽一起考虑，不能太长**

  所有代码块都必须使用 `{ }`，即使只有一行
  - 好：`if (cond) { return }`
  - 坏：`if (cond) return`
  - 例外：隐式返回的箭头函数 `() => expr`
- **箭头函数**：尽量优先使用箭头函数
  - 单参数时省略括号：`arg => { ... }`
  - 简单表达式优先隐式返回：`(a, b) => a + b`
- **错误处理**：抛错时不要写 `new`，用 `throw Error('message')`
- **默认值**：优先使用 `??`，不要滥用 `||`
- **避免 null**：优先使用 `undefined` 或直接省略属性
## 项目知识
- **配置规则同步**：`GEMINI.md` 是 `AGENTS.md` 的硬文件链接。因此只需专注更新本文件 `AGENTS.md`
- **代码行数统计**：所有关于代码行数（LOC）相关的评估，必须以执行 `./scripts/count-core-code.ps1` 的输出结果为准
- 在 Windows 环境涉及 `moon ide` 时，通过项目根目录的 `./mide.ps1` 调用，不要直接在 PowerShell 里运行 `moon ide ...`
## 测试相关 **重要**
- **所有测试**：运行 `./run test-all`，这是唯一正确的测试入口，可以确保测试到所有问题并且时序正确
- test-all 会并行进行三个分支 src/native/browser 的测试，非常强大而且快速
- 不要自己执行 moon check 或者 moon test，这只能检测 src 包里的内容（除非你就是想快速测试 src 包）
- 浏览器测试**不包含** service 的构建，所以一定要走 `./run test-all` ！！！！
- 测试阶段超时 5 秒已经绝对合理，**你还超过就是你的测试垃圾**，基本上来讲测试应该总是 1 秒全跑完
## 项目约束
- UI 操作和 CLI 操作必须走同一条底层命令路径，只允许触发位置不同，不允许语义和生命周期分叉
