# MetaEditor Rules
最高原则：功能相同的情况下，更短更精炼的代码就是更好的
新增代码前，先优先考虑能否用现有 utility 组合完成；如果需要引入新 utility，先判断它是否应当抽成可复用工具，避免在局部逻辑里重复造新层。
这不是单纯为了便于审查，而是好代码的核心标准之一。
## 协作规则
- 始终用中文回复用户
- 只改用户明确要求修改的内容，不顺手重排、重命名、整理或重构。
- 先保持现有文件的顺序、排版、紧凑度和命名风格，再考虑别的优化。
- 修改文件时，默认采用最小补丁，不整文件重写。
- 注释默认用中文。
- UI 操作和 CLI 操作必须走同一条底层命令路径，只允许触发位置不同，不允许语义和生命周期分叉。
- 禁止向项目目录及其任何子目录产生/遗留任何临时文件（如 debug_*.js, *.log, *.png 等）。
- 所有的测试产物（截图、日志）必须输出到系统的临时目录（如 $env:TEMP），读取后立即清理。
- 文档默认描述**当前状态**、当前约束和当前原则，不写“演进到”“未来再做”“之前是”这类需要结合历史版本理解的表述。
- 修改设计文档时，优先把现状、接口边界、系统原则写清楚；如果某项尚未完成，就直接写“当前未完成/当前边界”，不要写成路线图。
- 除非用户明确要求，避免在设计文档里引入带时间顺序的架构叙事；读者应只看当前文档就能理解系统。
- 修改 CLI 输出、命令说明或动作暴露方式时，默认遵守 `doc/meta-editor-service.md` 中的统一命令源与人类可读输出原则，不为 CLI 额外保留或新增平行 JSON 协议。
- 当用户要求“更新 devlog”时，默认需要更新 `doc/devlog.md`。
- 更新 `doc/devlog.md` 时，默认使用当前环境的本地日期和时段；如果用户没有指定日期或时段，就自动归档到当天对应的小节，如 `03-22 上午`、`03-22 下午`；若当天对应小节已存在，则直接续写，若不存在，则按现有文档风格创建。
- 续写 `doc/devlog.md` 时，先阅读相邻小节，保持上下文连贯、叙事密度和措辞风格一致。默认记录这次实际完成并验证过的工作、踩到的问题、最后收口的实现和测试变化，不写成任务清单。写 `doc/devlog.md` 时，要让读者只看日志本身就能理解，不要引用当前对话里的省略说法或代词。
## MoonBit 代码风格
- 禁止使用 moon format，和用户的目标风格极端不兼容
- 保持紧凑写法，不主动拉开空行。
- 状态 `let ...` 放在文件最开头。
- 类型定义放在下面，并和对应的类型操作放在一起。
- 类型标注写成 `x: T`，不要写成 `x : T`。
- 构造函数中的 `{{ ... }}` 按同一个符号风格保留，不要拆开整理。
- try catch 前面只有一个表达式时省略 try，直接 expr catch {…}
- 除非用户明确要求，不主动把短写法改成长写法。
## JavaScript 代码风格
- **不要分号**：语句结尾不要写分号
- **缩进**：使用 **2 个空格**
- **引号**：字符串使用单引号 `'`
- **行宽**：每行最多 **100** 个字符
- **大括号**：
  **最重要/最容易出错/要和行宽一起考虑，不能太长**

  所有代码块都必须使用 `{ }`，即使只有一行
  - 好：`if (cond) { return }`
  - 坏：`if (cond) return`
  - 例外：隐式返回的箭头函数 `() => expr`
- **箭头函数**：尽量优先使用箭头函数
  - 单参数时省略括号：`arg => { ... }`
  - 简单表达式优先隐式返回：`(a, b) => a + b`
- **错误处理**：抛错时不要写 `new`，用 `throw Error('message')`
- **默认值**：优先使用 `??`，不要滥用 `||`
- **避免 null**：优先使用 `undefined` 或直接省略属性

## 项目知识
- **代码行数统计**：所有关于代码行数（LOC）相关的评估，必须以执行 `scripts/count-core-code.ps1` 的输出结果为准。
- **回归测试的双层体系**：在对核心包（如 `service`）进行逻辑大改时，测试必须兼顾生命周期和宿主集成：
  1. **底层状态机与系统测试**：运行 `./test-native.ps1`（本质是构建并测试 `service` 包）。这主要运行 `server_test.mbt`，可以验证服务重连、JSON 解析、单例锁等无界面逻辑。
  2. **端到端集成与 App CLI 测试**：运行 `./meta.ps1 test`。由于此操作会直接使用最新代码构建并在后台运行真实的 `service.exe` 进程接管 CLI 请求，因此它是测试命令行进程分发、通信边界的唯一可靠验证方法（能捕获由于进程树、文件锁残留引起的死锁等在普通 `moon test` 中无法呈现的悬挂 Bug）。
- **配置规则同步**：`GEMINI.md` 是本项目中对于大模型配置文件的软链接并跟随 `AGENTS.md` 自动更新。因此维护相关规则时只需专注更新本文件 `AGENTS.md`。
