# MetaEditor 运行模型与逻辑布局讨论纪要

日期：2026-03-17 ~ 2026-03-18

## 1. 讨论主题

这份文档记录了一次围绕 `MetaEditor` 运行模型、多人协作、客户端/服务端分层、视图可测性、逻辑布局与 CSS 边界的连续讨论。整轮讨论不是线性推导，而是多次提出方案、发现冲突、修正前提、重新收敛的过程。

为了保留讨论轨迹，本文采用 `Codex / 用户` 的对话式纪要结构，而不是抽象总结式写法。这样做的目的，是让后续回看时可以直接看到：

- 某个判断最初是怎么提出来的
- 哪个反例推翻了它
- 为什么最后收敛到现在这条路线上

## 2. 第一个问题：metrics 到底该不该放在客户端

### 用户

用户首先意识到一个基础问题：如果有两个用户同时连接同一个 Core，那么他们渲染出来的 UI metrics 很可能不同。

这里的 metrics 不只是简单的节点宽高，还包括：

- `getBoundingClientRect`
- `clientWidth / clientHeight`
- `scrollTop / scrollLeft`
- viewport 中的可见性
- 选区像素区域
- 光标的屏幕坐标

用户的直觉是，这些结果明显受每个客户端自己的 viewport、滚动、字体渲染、DPR、容器尺寸等因素影响，因此最好放到客户端本地处理，而不是统一回传给服务端。

### Codex

Codex 对这个判断表示同意，并给出了一层更明确的解释：

- 共享的是语义状态
- 不共享的是投影结果

也就是说：

- 文档模型、业务状态、光标逻辑位置、选择范围等可以是共享的
- 但实际像素几何、滚动后的显示区域、可见性、布局测量等是 per-client 的本地事实

由此，Codex 最初自然推演出一套分层模型：

- 服务端持有共享 `state`
- 客户端持有本地 `local`
- 客户端根据 `state + local` 产生最终视图

当时这看起来是顺理成章的。

## 3. 用具体例子验证：`IntentDoc/dev/story/text.story.js`

### 用户

用户为了避免讨论停留在抽象层，直接给出一个现成例子：`IntentDoc/dev/story/text.story.js`，要求从这个例子出发分析，看看哪些和客户端测量相关的逻辑必须留在客户端本地执行。

### Codex

Codex 阅读该文件后，从中识别出一整组和本地环境强相关的逻辑，包括：

- `measureText`
- `buildVisualLineSegs`
- `getVisualRows`
- `findClosestCol`
- `findVisualRowByCursor`
- 光标 `x / y` 计算
- 自动滚动
- 指针坐标到文档位置的命中测试

进一步分析后指出，这些逻辑虽然看起来是“组件核心功能”，但其输入依赖的是客户端本地环境：

- 字体测量
- viewport 宽高
- 容器滚动
- DOM rect
- pointer 坐标
- computed style

所以这不是“浏览器 API 调用留在客户端、本体算法留在服务端”那么简单，而是整条“本地测量 -> 几何派生 -> 命中/滚动/可视结果”的链条都天然偏向客户端。

这一步把问题从“metrics 归属”推进到了“组件代码怎么分层”。

## 4. 第二个问题：组件到底该怎么拆，才不会把原有的简洁感拆没

### Codex

在确认 metrics 相关逻辑确实偏客户端之后，Codex 试图用一个比较常见的工程拆法去解释组件：

- `model`
- `projection`
- `host`
- `view`

这个说法的出发点是好的：试图把服务端逻辑、客户端逻辑和宿主实现拆清楚。

### 用户

用户马上指出，这种拆法虽然工程上容易说清楚，但会把原本组件的简洁感破坏掉。

用户强调，原先组件的核心体验其实是非常直接的：

- `state`
- `innerState`

这两个东西足够表达大多数组件的核心含义。继续抽成 `model / projection / host / view`，虽然概念上规整，却已经很难看出原本那种“一个组件整体”的自然写法了。

### Codex

Codex 接受这个批评，并开始从用户原有的组件模型内部去重新理解。

进一步分析发现，原本的 `innerState` 其实混着两种不同的东西：

第一类是组件自己关心的本地派生状态，例如：

- `cursor.x / cursor.y`
- `wrap`
- `viewportWidth / viewportHeight`
- `gutterWidth`
- visual row cache
- 本地 selection 的显示状态

第二类是纯宿主细节，例如：

- DOM refs
- canvas context
- IME session
- click session
- pointer drag session

于是 Codex 不再坚持 `model / projection / host / view` 作为主视角，而是收敛到一个更贴近原模型的解释：

- `state`：共享语义状态
- `local`：客户端本地派生状态
- `host`：宿主细节，保留在闭包里

这里还有一个关键修正：

- `host` 没必要成为一个显式输入参数
- 它本来就在组件内部以隐式闭包方式存在

这一点得到了用户的认可，因为它更接近原代码的感觉。

## 5. 第三个问题：代码到底怎么编译、怎么分发

### 用户

用户接着把问题推进到工程层面：

如果采用这种分层，那么代码到底该怎么定义、怎么编译、怎么分发？

同时用户提出一个很直接的工程判断：

- 直接编译成同一份 JS 源码
- 浏览器端实际上也持有服务端完整逻辑，只是不启用
- 这样分发非常简单

另外，用户还提出了一个关于更新方向的具体设想：

- 客户端向服务端发送更新 op
- 服务端返回给所有客户端具体更新

### Codex

Codex 认为这条路线非常现实，而且工程成本低。

在这个阶段，Codex 把这套模型初步整理为：

- 同一份组件源码
- 服务端运行时只启用 `state` 部分
- 客户端运行时启用 `state + local + view`
- `host` 保留为内部闭包状态

同时更新顺序统一为：

1. 客户端根据交互生成 op
2. 客户端发送 op 给服务端
3. 服务端应用 op 更新共享 `state`
4. 服务端广播共享状态变化
5. 客户端基于新的 `state + local` 重新生成视图

到这里为止，讨论暂时形成了一条非常顺滑的技术路线。

## 6. 关键转折：如果最终 view 下沉到客户端，AI 怎么观察最终视图

### 用户

就在这时，用户意识到一个更严重的问题：

最初引入这套无头 Core / projector 设计，是为了让 AI 能非常好地观察和测试视图层。

也就是说，Core 不只是持有 state，还应该持有足够可信、足够完整的 view 真相。

但如果按刚才那套模型走下去，最终 view 会被移到客户端。这样服务端虽然还有共享 `state`，却不再持有最终视图。

用户指出，这会直接破坏最初的目标。

### Codex

Codex 立刻承认这是一个真正的架构冲突，而不是小修小补能解决的问题。

为了挽救这个冲突，Codex 曾试图提出一个折中方案：

- 服务端保留 `semantic view`
- 客户端基于 `semantic view + local` 补成最终 `projected view`

这个方案的出发点是：

- AI 仍然可以在服务端看到语义视图
- metrics 等 per-client 部分仍然本地化

### 用户

用户进一步指出，这个折中并没有真正解决问题。

因为最初真正想要保住的，不只是“服务端还能看到某种 view”，而是：

**从最高层状态到实际最终视图之间的完整可测性。**

如果最终视图的关键部分仍然是在客户端补出来的，那么这条链路终究还是断了。

### Codex

Codex 接受这个反驳，并承认前面的 `semantic view / projected view` 方案虽然在分层上说得通，但在“最终可测性”这个目标上仍然失败。

这成为整轮讨论中最重要的一次否定。

## 7. 第四个问题：如果最终可测性优先，那是不是只能放弃多视图设计

### 用户

在确认“最终 view 下沉到客户端”不成立后，用户进一步提出更激烈的可能性：

如果继续保持“每客户端各自拥有自己的最终视图真相”，就会破坏从状态到视图的完整可测性。

在这种情况下，也许只能把“多视图、多客户端最终结果各自不同”的设计整体 drop 掉。

### Codex

Codex 没有直接反对，而是承认如果“完整可测性”是最高优先级，这个判断是完全可能成立的。

Codex 用更强的方式表述了这个结论：

- 如果要保证从最高层状态到最终视图结果的整条链都在 Core 内完整可观察、可测试、可审计
- 那么就不能把决定最终视图的关键逻辑下沉到客户端

这意味着：

- 多客户端可以连接
- 但不能让不同客户端拥有不同的最终视图真相

客户端最多只能承担非常薄的一层宿主行为。

这一步其实已经很接近重新回到最初更强的 headless-first 路线。

## 8. 用户提出关键判断：除非把布局本身提升到逻辑层

### 用户

就在讨论似乎要滑向“只能放弃多客户端本地差异”时，用户提出了一个关键判断：

> 除非我们直接把布局放到逻辑层做完

这句话把整轮讨论从“客户端分层怎么设计”一下子推进到更深的层面：

问题也许不是“view 在哪边生成”，而是：

**关键布局到底该由谁决定。**

### Codex

Codex 认为这是唯一真正能兼顾两边的路。

如果布局本身被提升到逻辑层，那么系统模型就会发生根本变化：

原本是：

- `state -> vnode -> DOM -> browser layout -> pixels`

现在变成：

- `state -> vnode + layout -> projector -> pixels`

也就是说：

- Core 不再只产出语义树
- Core 还要产出关键几何与布局结果
- 浏览器不再负责决定关键布局真相
- 浏览器只负责执行投影

这样一来，前面的冲突就出现了真正的解法：

- 最终关键视图仍然掌握在 Core
- AI 仍然能直接观察和测试它
- 客户端仍然可以存在，但只是 projector，而不是另一套共同拥有几何真相的运行时

## 9. 第五个问题：把布局放到逻辑层，是不是意味着要自己做一个浏览器

### 用户

用户接着把这个方向继续往前推，但没有直接要求完整浏览器布局，而是指出：

其实只要解决一些 UI 常见的简单布局，这套东西的可用性就会很强，剩下的样式可以交给 CSS。

这个判断很重要，因为它主动给“逻辑布局”设置了边界。

### Codex

Codex 认可这一判断，并进一步把边界讲得更清楚：

目标不是：

- Core 接管所有布局

而是：

- Core 接管关键几何
- CSS 继续负责非关键视觉样式

于是 Codex 明确区分：

Core 更适合负责的内容：

- 文本行/列布局
- 光标位置
- 选择区 rect
- gutter 宽度
- overlay / probe 几何
- 简单容器线性布局
- 虚拟列表 / 可视窗口切片
- 关键面板与编辑区分栏

CSS 继续负责的内容：

- 颜色
- 字体族
- 字号
- 边框
- 阴影
- 圆角
- 背景
- hover / transition
- 非关键 spacing 和装饰

在这个阶段，Codex 还额外指出了一组不应作为 `MetaEditor` 初始目标的复杂浏览器布局能力：

- 任意文档流混排
- 复杂 grid
- intrinsic sizing 黑魔法
- 完整 block / inline formatting context

原因是这些东西才是浏览器布局真正复杂的核心，一旦试图完整复刻，工程规模会迅速失控。

## 10. 用户追问：这些复杂浏览器布局能力到底复杂在哪里

### 用户

用户并没有直接接受“这几类东西很复杂”的说法，而是进一步要求把它们讲清楚。

### Codex

于是 Codex 逐项展开解释：

1. 任意文档流混排  
   指的是文本、inline 元素、图片、不同字号、不同基线元素自由混排时，浏览器需要处理换行、基线对齐、行高协调等复杂规则。

2. 复杂 grid  
   指 CSS Grid 完整能力下二维轨道之间复杂的尺寸协商、跨行跨列、内容反向决定轨道尺寸等问题。

3. intrinsic sizing 黑魔法  
   指 `min-content`、`max-content`、`fit-content`、`auto` 等由内容和容器相互决定尺寸的复杂规则。

4. 完整 block / inline formatting context  
   指浏览器默认文档排版规则，包括 margin collapsing、line box、float、clear、anonymous block boxes 等一整套体系。

解释完以后，Codex 再次强调：

- `MetaEditor` 并不需要也不应该完整复刻这些能力
- 它真正需要的是面向编辑器场景的一组受限布局原语

这一步的价值在于，逻辑布局方案到这里终于不再停留在口号，而开始具备明确的边界感。

## 11. 第六个问题：我们讨论的其实不只是 UI，而是一套多人协作运行逻辑

### 用户

在上述收敛之后，用户又意识到另一个概念层面的偏移：

这整套东西本质上已经不只是 UI 架构问题，而是在定义一套多人协作的交互逻辑。它和 CRDT 等协作算法的关系，需要单独说清楚。

### Codex

Codex 对这个判断表示赞同，并把几个概念拆开定位：

- CRDT / OT / CAS patch：协作算法层，解决共享数据如何收敛
- `state / local / host / view` 分工与运行模型：协作运行时层，解决多参与者围绕共享状态如何分工运行
- 文本 op、undo/redo、AI patch：交互语义层
- 布局、Probe、metrics、投影：视图与投影层

这一步的意义在于：

前面这整轮讨论并不是在替代 CRDT 之类的算法，而是在定义如果底层以后使用 CRDT、OT 或别的算法，上层的编辑器系统到底应该怎么运行。

## 12. 最终反转与当前方案

### 用户

在上述整轮推导之后，用户又向前走了一步，意识到一个更根本的反转：

前面之所以会不断遇到困难，是因为默认假设一直是：

- 一个无头 Core 持有主逻辑
- 浏览器只是 Projector
- CLI/AI 主要连到 Core

但更简单的路线，也许是把主运行时重新放回浏览器。

### Codex

Codex 起初把这个反转理解成一种工程上的简化：

- 浏览器本来就擅长文本测量、CSS、DOM、焦点、IME、滚动、选区
- 如果主逻辑回到浏览器，前面很多“跨运行时拆分”导致的问题会自然消失
- CLI 则退化成接到浏览器运行时的一个本地薄层

但这一阶段仍然存在一个潜在担忧：

- 如果浏览器重新成为主运行时，那么 CLI/AI 是否会失去对 UI 真相的直接访问？
- 是否会退回到截图、OCR、DOM selector 这种黑盒测试模型？

### 用户

用户随后指出，这里真正重要的一点是：

**CLI 仍然可以获取 UI 真相，而且拿到的是浏览器原生真相。**

新的方案并不是：

- 浏览器里跑应用
- CLI/AI 只能像传统自动化工具那样从外面黑盒观察

而是：

- 浏览器主运行时自己暴露结构化的 command / query / probe 接口
- CLI 通过本地薄层接到这个接口
- 因而 CLI/AI 仍然能进行结构化查询与全自动测试

这意味着：

- UI 真相的位置改变了
- 但 UI 真相没有丢失
- “不依赖截图 OCR”这一点依然成立

### Codex

Codex 在这一点上完全同意，并把它重新表述为：

- 旧模型：**单一 CLI / Core，多 UI / Projector**
- 新模型：**单一 UI 主运行时，多 CLI / 多控制端**

旧模型中：

- Core 是真相所在
- 浏览器是投影器
- 多个客户端围绕 Core 投影

新模型中：

- 浏览器运行时是主程序与 UI 真相所在
- CLI/AI/自动化工具都通过一个薄层接入浏览器运行时
- 浏览器对外暴露结构化 query / command / probe API

于是系统被重新理解为：

- **单一 UI 主运行时**
- **多 CLI / 多自动化入口**

而不是前面一直讨论的：

- **多 UI 视图**
- **单一 CLI / Core**

### 当前最终结论

这次反转之后，这个话题的最终方案可以概括为：

1. 暂时放弃“多 Projector 围绕单一无头 Core 投影”的主路线。
2. 改为“浏览器运行时承载完整编辑器逻辑与 UI 真相”的主路线。
3. CLI/AI 不再首先连接无头 Core，而是连接浏览器暴露出来的结构化接口。
4. 浏览器不是黑盒壳子，而是：
   - 主运行时
   - UI 真相持有者
   - command / query / probe 服务提供者
5. 因此，全自动测试、结构化 Probe 查询、AI 驱动操作仍然成立，而且拿到的是浏览器原生真相。
6. 这条路线的直接好处是：
   - 文本测量问题自然简化
   - CSS/样式系统自然贴着真实宿主实现
   - 组件模型不再被 Core / Projector 强拆
   - 工程复杂度显著下降

换句话说，这个话题最终不是收敛到“如何把 multiview 模型做对”，而是收敛到：

> **反转 multiview 设计，不做多 UI / 单 CLI，而做单一 UI / 多 CLI。**

## 13. 这份纪要的使用方式

这份文档保留了整轮讨论的推导过程，但服务的是最终方案，而不是中途那些已经被推翻的阶段性收敛。

如果以后只需要快速知道这轮讨论最后落在什么方案上，应直接看第 12 节。

如果以后需要知道“为什么前面某个看起来合理的方案最后被放弃了”，应该从头按 `Codex / 用户` 的往复过程阅读，因为真正重要的不是某个静态结论，而是这些结论为什么一步步变成现在这样。
