# MetaEditor 项目演变总结

这份文档不按时间复述开发日志，只提炼当前这段开发里真正有价值的演变脉络。
重点是三件事：

- 哪些主模型后来被确认是对的
- 哪些方向认真试过以后被收缩或放弃
- 当前项目的心智为什么会收成现在这样

## 1. 总体方向

这一段时间里，项目最稳定的一条主线不是“不断加功能”，而是反复做两类判断：

- 删掉并行路径，只留一条主路径
- 删掉多余概念，只保留能解释真实语义的那层

很多表面上看像功能开发的内容，真正的价值都在这里。`service`、`storage`、DOM、
bridge、测试、自动化，这几条线最后都在往这个方向收。

## 2. service：从并行原型收成单一路径

项目最早同时有旧 CLI、native websocket 原型、浏览器侧代码和 service 方向文档。
后面的关键变化，不是再补一层入口，而是把旧 `cli/` 和 `native_ws/` 逐步退出主线，
让 `service/` 接管真正的入口和生命周期。

围绕这条线，`meta start / stop / status`、浏览器握手、重连、host stop、页面 ready
和 native lifecycle test 也被逐步收成同一条可信路径。这里真正建立起来的，不只是
一个能跑的命令行工具，而是“MetaEditor 就是长期运行的 native service”这件事。

这条线里还有一次很重要的继续收缩。service 曾经一度往 host app、多 app、query、
exec、history、app test 调度这些方向长过，后面判断这批东西在当时都太重，于是又
连续做减法，把主模型压回最小 host 页面和最基本的生命周期能力。这个收缩很关键，
因为它把“想做很多宿主能力”和“当前真正能站稳的主路径”重新分开了。

## 3. storage：从中心化图模型退回节点级语义

`storage` 是这段时期里演变最剧烈的一条线。

一开始，代码曾经围着 `PersistValue`、graph、`GraphValue` 这套中心化图模型来回改，
试图用一套统一表示同时承载运行时对象图和持久化格式。这个方向表面上很完整，
但后来越来越清楚，它和原始 JS 模型的心智其实并不一致。

真正的转折点，是重新对齐 `storage.js` 以后确认：原始模型并不是“从根遍历整图再同步”，
而是“每个对象节点自己 watch 自己，再做浅写盘”。一旦这一点重新说清楚，前面那条
graph 路线就不再是“还有几处细节没补”，而是主模型本身偏了。

后面因此不是给 graph 再打一层补丁，而是整条中心化图同步路线被删掉，重新收成
节点级 storage。运行时对象、子节点引用、对象身份复用、浅写盘、按节点同步，这些
语义才重新回到正确位置。这里最重要的变化不是类型名替换，而是 storage 的心智换了。

## 4. storage 边界：stop 不删盘，gc 独立处理

节点级 storage 立起来以后，真正反复讨论的重点很快从“能不能跑”变成“边界到底是什么”。

其中最关键的一次收束，是把 `stop()` 和删盘语义彻底拆开。storage 只负责停止同步，
不再把节点解绑、作用域结束或父引用变化误当成删盘时机。磁盘上的孤儿节点保留，
后面再由显式 `gc_refs()` 处理。

这条边界很重要，因为它把三件事重新分开了：

- 内存对象的生死交给运行时
- storage 只管“还同步不同步”
- 持久化层的垃圾清理由独立 gc 处理

这类判断比“又补了一条同步逻辑”更值钱，因为它决定了后面整个 storage 不会继续在
错误的职责边界上长。

## 5. UI / DOM / 测试：重新回到同一套运行时语义

UI 这条线的价值，也不在于“又多了几条测试”，而在于测试、bridge、自动化和运行时
最后开始共享同一套语义。

中间曾经试过额外做一层测试 runtime，想把 `mount / click / text 查询` 这些操作压短。
后面很快确认这条线会在测试层重复维护一份 DOM 语义，于是又把它删掉，白盒测试重新
回到 `dom.mbt` 自己的 `realize / get_cmds / flush` 这条主路径。

同一时期，节点定位也从 `data-testid` 收成正式的 `ui-id`。这不只是属性名变化。
它意味着节点标识不再被理解成“测试专用辅助字段”，而是正式运行时会认的一层稳定语义。
事件分发也跟着从内部 callback id 收成 `ui-id + 事件名`，于是浏览器 bridge、host
自动化、白盒测试和运行时事件回传，终于开始走同一条底层路径。

## 6. UI 查询模型：从平面标识继续长成作用域路径

`ui-id` 这条线后面没有停在“统一属性名”，而是继续往真正的查询模型发展。

后面的重点逐步变成：

- 普通组件默认不带名字域，需要时才显式开
- 名称域和列表域分开，不再混成一种东西
- `h_map` 的查询语义也对齐到同一套列表路径

这一步的价值，在于节点查询不再只是平面的字符串匹配，而是开始有“组件作用域 /
列表项 / 动态项”这些更接近真实 UI 结构的路径语义。这样以后，查询、事件分发、
浏览器回传和自动化操作才有机会长期共用同一套模型。

## 7. wasm / WAMR：认真试过，最后明确放弃

这段时间里，wasm 路线并不是随便看了一眼，而是认真做过一轮实验。包括 WAMR 最小接入、
真实 MoonBit wasm 产物、宿主调用、实例状态、宿主回包，以及 browser_ui 能力模型，
都已经试到了足够说明问题的程度。

这条线后来的结论不是“暂时没做完”，而是明确放弃。原因也不是单点技术障碍，而是整条
路线涉及的宿主接口、能力模型、运行时形态和工程复杂度综合下来，都不值得继续压在当前
主线上。

这段试验仍然有价值，因为它逼着项目把 managed app 的宿主能力边界想清楚了一次。
但它的价值停在“帮助判断方向”，不再进入当前主线。

## 8. 工程方法：先做最小实验，再决定保留、收缩还是删除

如果要用一句话概括这段开发过程，最合适的说法其实不是“功能逐步丰富”，而是：
项目不断用最小实验来逼近真正的主线。

这里面有几种反复出现的方法：

- 先做最小可运行版本，再决定这条线值不值得继续长
- 一旦确认主模型偏了，就直接删掉，不为旧抽象保留平行入口
- 测试尽量回到真实运行时语义，不额外发明测试专用模型
- 文档不追求把所有过程都记下来，而是沉淀已经确认的边界和结论

所以这段时期真正留下来的成果，不只是若干功能点，而是项目逐步形成了一种很清楚的工作方式：
先试，尽快判断，能收就收，不能收就删，把主路径越压越窄，越压越准。
