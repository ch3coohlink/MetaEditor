# MetaEditor 服务方案

本文档描述 MetaEditor 作为**持续运行的元编辑器服务**时的职责、边界与接口原则。这里的 `service` 不是旁路组件；它就是 MetaEditor 主程序本身，负责管理 app 项目的编译、运行、浏览器页面会话，以及对浏览器页面的结构化查询与操作。该方案的主题是让整个控制面、测试面与服务面尽量由 MoonBit 承担。

## 1. 目标

- 让浏览器自动化成为系统内生能力，而不是依赖外部 JavaScript 测试框架。
- 让项目编译、运行、页面连接、UI 查询、UI 操作进入同一个服务模型。
- 让 CLI、服务主逻辑、测试主逻辑尽量完全由 MoonBit 承担。
- 让 JavaScript 只保留无法脱离浏览器宿主 API 的最小桥接代码。
- 让测试表达建立在“真实浏览器页面 + 结构化 query / exec”之上，而不是截图、OCR 或外部脚本推断。

## 2. 服务定位

MetaEditor `service` 是一个长期驻留进程。它持续维护项目状态与浏览器页面状态，而不是每次执行命令时临时启动一套新环境。

服务的核心职责如下：

- 管理一个或多个 app 项目的编译、启动、停止。
- 维护每个项目的浏览器测试页面会话。
- 为 CLI、脚本、自动化测试提供统一控制入口。
- 通过浏览器桥获取浏览器内部结构化真相，并执行真实浏览器操作。

`service` 不等于某个具体业务 app；它是 MetaEditor 主程序，是 app 之上的宿主与控制层。

这里的“服务”默认指 MoonBit 主程序。JavaScript 只保留浏览器宿主适配作用，不承载系统策略。

## 3. 最小职责

第一版服务只要求覆盖以下闭环：

- 注册一个 app 项目。
- 触发该项目的编译。
- 启动该项目。
- 确保存在一个可用的测试专用页面；若不存在则打开一个。
- 通过该页面执行 query / exec / action。
- 通过该页面执行 `test`，完成当前 app 的结构化闭环验证。
- 输出结构化结果给 CLI 或测试调用方。

以下职责不属于第一版最小范围：

- 多项目并发调度策略。
- 页面池、多标签页编排。
- 权限域、动作域、多控制端协作。
- 分布式执行或远程设备管理。

## 4. 页面模型

服务使用**真实浏览器页面**作为测试与控制目标，而不是内嵌浏览器模拟器。

页面模型遵循以下规则：

- 每个项目有一个测试专用页面概念。
- 该页面优先复用，而不是每次测试都重新创建。
- 当测试专用页面不存在、失效或连接中断时，服务负责重新打开页面并恢复连接。
- 页面连接成功后，服务通过浏览器桥读取页面内部结构化状态。

这里的页面不是“测试工具生成的临时页面”，而是项目真实运行页面的一种受控会话。

## 5. 浏览器桥职责

浏览器桥是浏览器宿主适配层，只负责必须由浏览器 API 完成的工作：

- 建立与服务的连接。
- 执行 DOM 指令。
- 监听浏览器事件并回传。
- 读取浏览器原生状态，如节点结构、文本、属性、焦点、几何、可见性、视口信息。
- 执行真实浏览器操作，如 click、focus、input、scroll。

浏览器桥不负责：

- 测试调度。
- 项目编译与运行管理。
- 复杂协议决策。
- 外部测试框架语义。

浏览器桥的存在不表示系统以 JavaScript 为主；它只是浏览器宿主适配层。

## 6. 测试模型

MetaEditor 的测试模型建立在服务能力之上。

测试的基本形态是：

1. 确保目标项目已编译并运行。
2. 确保测试专用页面已连接。
3. 对页面执行结构化 `query`。
4. 对页面执行结构化 `exec` 或 `action`。
5. 再次执行 `query`。
6. 对结构化结果做断言。

因此测试表达应尽量写成：

- `ensure_project_ready`
- `ensure_page`
- `test`
- `query ui`
- `exec click`
- `query focused`
- `assert ...`

而不是写成：

- 外部 JavaScript harness 启动服务
- 浏览器测试库驱动页面
- 外部框架推断 UI 结果

## 7. 当前测试入口

- `service` 自身的局部逻辑通过 `moon test` 验证。
- 当前 app 的完整浏览器闭环通过运行中的 `service` 执行 `test` 命令验证。
- 正式测试路径不再依赖 JavaScript smoke harness。

## 8. MoonBit 与 JavaScript 的边界

该方案要求尽量减少 JavaScript 在系统中的占比，并把“能用 MoonBit 就用 MoonBit”作为默认决策原则。

边界定义如下：

- **MoonBit 负责**：
  - CLI 与服务主逻辑
  - 项目管理
  - 会话管理
  - query / exec / action 协议
  - 状态机
  - 测试组织与断言逻辑
  - 编译、运行、页面管理等服务调度逻辑
- **JavaScript 负责**：
  - 浏览器宿主 API 适配
  - 必须直接调用 DOM / WebSocket / `localStorage` 的部分

CLI 与服务端当前都使用 MoonBit native 端，Node.js 不再承担项目运行时职责。

## 9. 接口原则

服务对外至少暴露以下能力：

- 项目注册
- 项目编译
- 项目启动
- 页面确保在线
- 页面状态查询
- 页面动作执行
- 当前 app 测试执行
- 会话状态读取

接口原则遵循以下约束：

- CLI 默认输出必须面向人类可读，不为 CLI 单独维护一套平行的 JSON 状态协议。
- 生命周期命令如 `start`、`stop`、`test` 可以输出最小必要的手写文本，因为这些结果不直接来自 host app 视图。
- UI 状态相关命令必须优先直接复用 host app 已经整理好的视图，而不是再定义一套独立的状态输出格式。
- 命令说明、UI 操作、CLI 操作必须来自同一个动作源头；按钮标题、命令说明和 CLI 可执行动作不能各自维护独立文案。
- 如果某个信息已经在 UI 视图里以更自然的形式存在，就不应再为 CLI 额外发明一层结构化状态描述。

## 10. 当前关键约束

- 浏览器页面必须是可复用会话，而不是每次测试都新建匿名页面。
- 查询结果必须来自浏览器内部真实状态。
- 操作必须作用于真实页面，而不是测试框架中的影子对象。
- 测试能力必须复用系统自己的 query / exec 协议。
- 服务必须长期运行，才能承担项目状态和页面状态的统一管理。
- CLI 与服务主逻辑默认使用 MoonBit native 端，不能把关键状态机继续留在 JavaScript 中。

## 11. 当前未决问题

以下问题需要后续继续设计，但不影响本方案成立：

- 项目编译/运行任务的具体调度模型。
- 服务如何打开真实浏览器页面。
- 多项目场景下的资源隔离与命名规则。
- 多 app 测试如何组织与选择目标 app。
