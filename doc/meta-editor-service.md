# MetaEditor 服务方案

本文档描述 MetaEditor 作为**持续运行的元编辑器服务**时的职责、边界与接口原则。这里的服务不是一次性测试脚本，也不是外部测试框架的包装层，而是系统自身的一部分：它负责管理 app 项目的编译、运行、浏览器页面会话，以及对浏览器页面的结构化查询与操作。该方案的主题不只是替代 Playwright，而是让整个控制面、测试面与服务面尽量由 MoonBit 承担。

## 1. 目标

- 让浏览器自动化成为系统内生能力，而不是依赖 Playwright 一类外部测试框架。
- 让项目编译、运行、页面连接、UI 查询、UI 操作进入同一个服务模型。
- 让 CLI、服务主逻辑、测试主逻辑尽量完全由 MoonBit 承担。
- 让 JavaScript 只保留无法脱离浏览器宿主 API 的最小桥接代码。
- 让测试表达建立在“真实浏览器页面 + 结构化 query / exec”之上，而不是截图、OCR 或外部脚本推断。

## 2. 服务定位

MetaEditor 服务是一个长期驻留进程。它持续维护项目状态与浏览器页面状态，而不是每次执行命令时临时启动一套新环境。

服务的核心职责如下：

- 管理一个或多个 app 项目的编译、启动、停止。
- 维护每个项目的浏览器测试页面会话。
- 为 CLI、脚本、自动化测试提供统一控制入口。
- 通过浏览器桥获取浏览器内部结构化真相，并执行真实浏览器操作。

服务本身不等于业务 app；它是 app 之上的宿主与控制层。

这里的“服务”默认指 MoonBit 主程序，而不是 Node.js 逻辑层。服务端如果需要 JavaScript，也只允许保留宿主适配作用，不承载系统策略。

## 3. 最小职责

第一版服务只要求覆盖以下闭环：

- 注册一个 app 项目。
- 触发该项目的编译。
- 启动该项目。
- 确保存在一个可用的测试专用页面；若不存在则打开一个。
- 通过该页面执行 query / exec / action。
- 输出结构化结果给 CLI 或测试调用方。

以下职责不属于第一版最小范围：

- 多项目并发调度策略。
- 页面池、多标签页编排。
- 权限域、动作域、多控制端协作。
- 分布式执行或远程设备管理。

## 4. 页面模型

服务使用**真实浏览器页面**作为测试与控制目标，而不是内嵌浏览器模拟器。

页面模型遵循以下规则：

- 每个项目有一个测试专用页面概念。
- 该页面优先复用，而不是每次测试都重新创建。
- 当测试专用页面不存在、失效或连接中断时，服务负责重新打开页面并恢复连接。
- 页面连接成功后，服务通过浏览器桥读取页面内部结构化状态。

这里的页面不是“测试工具生成的临时页面”，而是项目真实运行页面的一种受控会话。

## 5. 浏览器桥职责

浏览器桥是浏览器宿主适配层，只负责必须由浏览器 API 完成的工作：

- 建立与服务的连接。
- 执行 DOM 指令。
- 监听浏览器事件并回传。
- 读取浏览器原生状态，如节点结构、文本、属性、焦点、几何、可见性、视口信息。
- 执行真实浏览器操作，如 click、focus、input、scroll。

浏览器桥不负责：

- 测试调度。
- 项目编译与运行管理。
- 复杂协议决策。
- 外部测试框架语义。

浏览器桥的存在不表示系统以 JavaScript 为主；它只是浏览器宿主适配层。

## 6. 测试模型

MetaEditor 的测试模型建立在服务能力之上。

测试的基本形态是：

1. 确保目标项目已编译并运行。
2. 确保测试专用页面已连接。
3. 对页面执行结构化 `query`。
4. 对页面执行结构化 `exec` 或 `action`。
5. 再次执行 `query`。
6. 对结构化结果做断言。

因此测试表达应尽量写成：

- `ensure_project_ready`
- `ensure_page`
- `query ui`
- `exec click`
- `query focused`
- `assert ...`

而不是写成：

- 打开 Playwright
- 查找页面元素
- 截图
- OCR
- 外部框架推断 UI 结果

## 7. 与 Playwright 的关系

当前方案的目标有两个：

- **移除 Playwright 作为主测试框架**
- **让控制逻辑、测试逻辑、服务逻辑尽量完全 MoonBit 化**

服务直接控制真实浏览器页面，浏览器桥直接暴露结构化 query / exec，因此系统不再依赖 Playwright 提供“浏览器自动化主能力”。

如果某些极少数场景仍需要外部浏览器工具，它们也只应作为临时宿主辅助，而不是系统主测试入口。

## 8. MoonBit 与 JavaScript 的边界

该方案要求尽量减少 JavaScript 在系统中的占比，并把“能用 MoonBit 就用 MoonBit”作为默认决策原则。

边界定义如下：

- **MoonBit 负责**：
  - CLI 与服务主逻辑
  - 项目管理
  - 会话管理
  - query / exec / action 协议
  - 状态机
  - 测试组织与断言逻辑
  - 编译、运行、页面管理等服务调度逻辑
- **JavaScript 负责**：
  - 浏览器宿主 API 适配
  - 必须直接调用 DOM / WebSocket / `localStorage` 的部分

对于 CLI 与服务端，目标状态是使用 MoonBit native 端，而不是继续依赖 Node.js 作为逻辑主体。Node.js 如仍存在，只能作为过渡性的宿主外壳。

## 9. 接口原则

服务对外至少暴露以下能力：

- 项目注册
- 项目编译
- 项目启动
- 页面确保在线
- 页面状态查询
- 页面动作执行
- 会话状态读取

所有接口返回结构化结果，而不是仅返回打印文本。

## 10. 当前关键约束

- 浏览器页面必须是可复用会话，而不是每次测试都新建匿名页面。
- 查询结果必须来自浏览器内部真实状态。
- 操作必须作用于真实页面，而不是测试框架中的影子对象。
- 测试能力必须复用系统自己的 query / exec 协议。
- 服务必须长期运行，才能承担项目状态和页面状态的统一管理。
- CLI 与服务主逻辑默认使用 MoonBit native 端，不能把关键状态机继续留在 JavaScript 中。

## 11. 当前未决问题

以下问题需要后续继续设计，但不影响本方案成立：

- 服务与浏览器之间由谁监听 WebSocket。
- 项目编译/运行任务的具体调度模型。
- 服务如何打开真实浏览器页面。
- 多项目场景下的资源隔离与命名规则。
- 如何把现有测试逐步迁移到服务内生测试模型。
